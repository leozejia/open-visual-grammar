# Case: Proof Screenshot

## Job

Test whether a proof-led visual should become a stable pattern or remain an
incubation note.

## Typical pattern

No stable pattern yet. Use this case to evaluate a proof or screenshot-led
candidate.

## Scenario

A tutorial depends on a concrete visible state:

```text
The reader needs to see the difference between a confusing UI state and the
actual successful action path.
```

## Required score fields

- Viewer
- Job
- Proof needed
- Real screenshot source
- Annotation model
- Density
- Privacy boundary
- Forbidden

## Runtime target

Real screenshot plus deterministic annotation, or generated cover paired with a
real inline screenshot.

## Three-card requirement

The three candidates should test different proof strategies:

- direct before/after;
- one screenshot plus callout;
- conceptual cover plus real inline proof.

Do not ask an image model to invent exact product UI.

## Pass signals

- The reader can act with more confidence.
- Private data is removed.
- The annotation helps instead of decorating.
- The visual is honest about what is real and what is illustrative.

## Fail signals

- Fabricated UI proof.
- Too many arrows or boxes.
- Private account, path, token, or customer data.
- Screenshot is accurate but unreadable at target size.
